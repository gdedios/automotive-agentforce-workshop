# BLOCKER: Phase 6b Data Library upload fails — root cause identified

**Status:** ROOT CAUSE FOUND — this Atlas trial does NOT have Data 360 / Data Cloud provisioned.
**Org:** Electra_Auto (`epic.c6fb0dbb14e4@orgfarm.salesforce.com`, instance `00DgK00000Q8aKJUAZ`)
**Date:** 2026-05-27

## Root cause (CONFIRMED)

Salesforce help docs (`ai.data_library_custom_retriever`):
> **"To use Agentforce Data Libraries, you must have Data 360 set up. Data libraries
> consume Data 360 credits."**

Probes against this org confirm Data 360 is OFF:
- `OrganizationType` = **Developer Edition** (not a true Atlas trial — orgfarm
  Atlas pool actually delivers DE orgs with the Agentforce add-on toggled on)
- `GET /services/data/v66.0/connect/adl/data-libraries` → 404 NOT_FOUND
  (endpoint doesn't exist on orgs without Data 360)
- `GET /services/data/v66.0/connect/uma/dataSpaces` → `FUNCTIONALITY_NOT_ENABLED:
  [MarketingConnect]`
- `GET /services/data/v66.0/ssot/...` → 404 (Data Cloud REST surface not registered)

The ADL UI in Setup is **rendered shell only** — Salesforce shows the wizard, lets
you create a library row and stage files, but the ingest backend belongs to Data 360.
With Data 360 OFF, the Lightning UI submits the file and the backend rejects with
the generic "We couldn't upload your file. Please create a new data library and try
again." That instruction text is misleading; deleting and recreating the library
will never fix it on this org type.

## Likely root causes (ranked)

1. **OrgFarm trial provisioning gap.** `AiAuthoringBundle` works (we deployed an agent
   to this org in Phase 5), but ADL file ingestion may rely on a separate backend
   service that isn't fully provisioned for fresh orgfarm pulls. No public docs on this.
2. **Permission gap.** The `epic.c6fb0dbb14e4` user is the org admin, but the ADL
   upload may need `Manage Data Library` or a Content Pack permission that isn't
   inherited by the system admin profile in trials.
3. **Content Asset / ContentVersion storage gate.** Some orgs need
   "Salesforce CRM Content" enabled in `Salesforce CRM Content Settings`.
4. **API blocking.** Atlas trial may rate-limit the ADL ingest endpoint differently
   than the catalog endpoint we used for the agent deploy.
5. **Library #2 is genuinely broken just like #1.** Then a third try might succeed.

## Updated path forward (after root cause confirmed)

Options A (retry UI) and B (REST API) are now ruled out — both fail at the same
backend layer. Remaining options:

### B′. Provision Data 360 on this org (most pedagogical)
Atlas trial may include a Data 360 entitlement that just needs to be enabled in
Setup. From Setup → Data Cloud Setup or Data 360 Setup, run the activation wizard
(may require accepting terms + a few minutes of provisioning). If the SKU is
actually present, this unblocks UI + REST in one step. **Risk:** orgfarm Atlas
trials may not include Data 360 SKU at all → wizard rejects.

### C. Ask in #orgfarm
If B′ has no activation wizard, ask Slack `#orgfarm-orgs` whether Data 360 can be
enabled on Atlas trials, or whether a Data-Cloud-included pool exists.

### D. Switch to a Data-360-bundled pool org
Orgs with `Atlas + Data Cloud` may exist in a different orgfarm pool. Reload from
that pool. **Cost:** Phase 1-5 work has to be redeployed to the new org (the
package.xml + scripts make this ~30 min, not a full rebuild).

### E. Ship workshop without RAG / AQWK
Estado_y_FAQ subagent works today with FAQ instructions baked into reasoning text.
Document the limitation and proceed to Phase 7 / 8. **Cost:** workshop loses RAG —
a major Agentforce capability students should see.
